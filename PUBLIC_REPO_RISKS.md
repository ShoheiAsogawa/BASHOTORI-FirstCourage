# リポジトリを公開にするときのリスクと対策

BASHOTORI を **Public** にする前に確認したい点です。

---

## 公開しても問題ない点（現状）

| 項目 | 状態 |
|------|------|
| `.env` | `.gitignore` に含まれており、コミットされない |
| キーの参照 | コードは `import.meta.env.VITE_*` のみ使用（ハードコードなし） |
| 本番のキー | GitHub Secrets 等で管理（リポジトリには含めない） |
| Gemini API キー | 公開フロントに埋め込む場合は Google 側のキー制限を必ず設定する |

---

## 公開時の主なリスク

### 1. キーが過去のコミットに入っている場合

- **リスク**: 昔 `.env` をコミットした、またはドキュメントに本物のキーを書いてコミットした場合、**履歴に残り続ける**。
- **対策**: 公開前に「キーが含まれるファイルを履歴から消す」か、少なくとも **現在のファイルに本物のキーが書かれていないか** を確認する。

### 2. Supabase の anon key について

- **VITE_SUPABASE_ANON_KEY** はフロントに埋め込まれるため、公開サイトのソースを見れば理論上は取り出せる。
- Supabase の **anon key は「公開前提」** で設計されており、**RLS（Row Level Security）と認証** でデータを守る想定。
- リポジトリを Public にしても「anon key がコードで読める」こと自体は、本番ビルドと同等。**RLS とポリシーが正しくかかっていれば、公開リポジトリにしたからといって追加リスクは小さい**。

### 3. ビジネスロジック・仕様の公開

- コードが読まれるため、**画面の動き・データ構造・API の使い方** は第三者に知られる。
- 機密性の高い業務ロジックや未公開機能がある場合は、公開を避けるか別リポジトリで非公開にする選択肢がある。

### 4. 依存関係の脆弱性

- `package.json` が公開され、依存ライブラリが把握される。**悪用されうる既知の脆弱性** があるとターゲットにされやすくなる。
- 対策: `npm audit` で脆弱性を確認し、修正してから公開する。

## キーを検索する方法

### 方法1: スクリプトで一括検索（推奨）

プロジェクトルートで実行します。

**Windows（PowerShell）:**
```powershell
.\scripts\check-secrets.ps1
```

**Mac / Linux（bash）:**
```bash
bash scripts/check-secrets.sh
```

キーっぽいパターン（`sb_secret_`、JWT、`AIzaSy`、Supabase URL など）が含まれるファイルを表示します。  
`xxxxx` のようなプレースホルダーにヒットすることもあります。**本物のキーが書かれているファイルだけ修正**すればOKです。

---

### 方法2: 手動で検索

**VS Code / Cursor で:**
1. `Ctrl+Shift+F`（検索を開く）
2. 次を順に検索する（正規表現にチェック）:
   - `sb_secret_[A-Za-z0-9_-]{20,}` … Supabase の service_role キー
   - `AIzaSy[A-Za-z0-9_-]{30,}` … Google API キー
   - `eyJ[A-Za-z0-9_-]{50,}` … JWT（anon key など）
   - `https://[a-z0-9]{10,}\.supabase\.co` … 本物の Supabase URL（短い `xxxxx` はプレースホルダーの可能性）

**ターミナルで（ripgrep がある場合）:**
```bash
rg "sb_secret_|AIzaSy[A-Za-z0-9_-]{30,}|eyJ[A-Za-z0-9_-]{50,}" --type-not lock
```
`.git` と `node_modules` は自動で除外されることが多いです。

---

### 方法3: .env が Git 履歴に入っていないか確認

```bash
git log -p --all -- .env
```
何も出かなければ、`.env` は履歴に含まれていません。

```bash
git log --oneline --all -- "*.env" ".env"
```
`.env` を変更したコミットの一覧が出ます。出た場合は、そのコミットで本物のキーが追加されていないか確認してください。

## 公開前にやることチェックリスト

- [ ] **現在のファイル** に本物のキー・パスワード・トークンが書かれていないか検索する（`sb_secret_`、`AIzaSy`、Supabase URL など）。
- [ ] **`.env` がコミットされていないか** 確認する（`git log -p -- .env` などで履歴を確認）。
- [ ] 本番で使うキーは **GitHub Secrets** 等の安全な場所のみで設定し、リポジトリには一切含めない。
- [ ] Supabase の **RLS とストレージポリシー** が有効で、認証済みユーザー以外は適切に制限されているか確認する。
- [ ] （推奨）`npm audit` で重大な脆弱性がないか確認し、可能なら直してから公開する。

---

## まとめ

- **キーをリポジトリに含めず、Secrets/環境変数のみで管理している** なら、公開による「キー漏洩」のリスクは小さい。
- **Supabase anon key** は公開前提の設計なので、RLS がかかっていれば公開リポジトリにしても大きな追加リスクにはならない。
- 公開で増える主なリスクは **「コード・仕様が読まれること」** と **「過去のコミットにキーが入っていないか」**。  
  上記チェックリストを満たしてから Public にすれば、GitHub Pages 用に公開するリスクは抑えられる。
