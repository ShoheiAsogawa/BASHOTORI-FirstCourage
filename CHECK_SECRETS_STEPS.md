# キー検索スクリプトの詳細手順（方法1）

スクリプトでリポジトリ内のキー・秘密情報を検索する手順です。

---

## 前提

- **Windows** で **PowerShell** を使います。
- プロジェクト（BASHOTORI）をクローンしたフォルダまで開いておいてください。

---

## ステップ1: プロジェクトのルートに移動する

1. **PowerShell** を開く  
   - スタートメニューで「PowerShell」と入力 → **Windows PowerShell** を起動  
   - または Cursor の **ターミナル**（`` Ctrl+` ``）で、シェルが PowerShell になっていることを確認

2. プロジェクトのルート（BASHOTORI フォルダ）に移動する  
   ```powershell
   cd C:\Users\hoikuen\BASHOTORI
   ```  
   ※ 実際のパスが違う場合は、BASHOTORI があるフォルダに合わせて変更してください。

3. 今いる場所を確認する  
   ```powershell
   pwd
   ```  
   - 表示の最後が `BASHOTORI` になっていればOKです。  
   - ここに `scripts` フォルダと `package.json` があるはずです。

---

## ステップ2: スクリプトの実行を許可する（初回のみ）

PowerShell では、初めてこのスクリプトを実行するときに「実行ポリシー」で止まることがあります。

1. 次のコマンドを実行する  
   ```powershell
   Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser
   ```  
2. 質問が出たら **Y** を入力して Enter  
3. これで「現在のユーザー」だけ、ローカルのスクリプト（`.ps1`）を実行できるようになります。

※ すでにスクリプトが実行できる環境なら、このステップは飛ばしてOKです。

---

## ステップ3: スクリプトを実行する

プロジェクトのルート（BASHOTORI）にいる状態で、次を実行します。

```powershell
.\scripts\check-secrets.ps1
```

- `.\` は「カレントフォルダの」という意味です。  
- パスが違うと「スクリプトが見つからない」と出るので、そのときは **ステップ1** で `cd` のパスを確認してください。

---

## ステップ4: 結果の見方

### 何もヒットしなかった場合

- 「本物のキーっぽい文字列は見つかりませんでした」と表示されます。  
- 「.env は履歴に含まれていないようです」と出ていれば、その部分も問題ありません。  
→ **このまま公開してよい状態**です。

---

### 何かヒットした場合

スクリプトは次のような「キーっぽいパターン」にヒットします。

| 表示されるパターン例 | 意味 | 対処 |
|----------------------|------|------|
| `sb_secret_...` | Supabase の service_role キー | **必ず** 該当ファイルを開き、本物のキーなら削除するか `sb_secret_xxxxx` に置き換える |
| `eyJ...`（長い文字列） | JWT（anon key など） | 本物なら `eyJxxxxx` などプレースホルダーに置き換える |
| `AIzaSy...` | Google API キー | 本物なら `AIzaSyxxxxx` に置き換える |
| `https://〇〇〇.supabase.co` | Supabase の URL | `〇〇〇` が本物のプロジェクトIDなら `https://xxxxx.supabase.co` に置き換える。`xxxxx` だけの場合は例なのでそのままでOK |
| `AKIA...` | AWS Access Key ID | 本物なら削除するかプレースホルダーに置き換える |
| `sk-...` | その他 API キー | 本物なら削除するかプレースホルダーに置き換える |

**やること:**

1. 表示された **ファイルパス** をメモする（例: `.\GITHUB_PUBLIC_REPO_CHECK.md`）
2. そのファイルを Cursor で開く
3. 検索（Ctrl+F）で `sb_secret_` や `AIzaSy` などを入力し、該当箇所を探す
4. **本物のキー** が書いてあれば、その部分を削除するか `xxxxx` のようなプレースホルダーに書き換える
5. 保存してから、再度 **ステップ3** のスクリプトを実行し、同じファイルがもう出てこないか確認する

---

### 「.env が履歴に…」と出た場合

- 「.env または VITE_SUPABASE を含むファイルが履歴に存在する可能性があります」と表示された場合は、**過去のコミット** に `.env` やキーが含まれている可能性があります。
- その場合は、表示されたコミット（例: `abc1234 メッセージ`）をメモし、  
  - 履歴からキーを消す（BFG や `git filter-branch` など）か、  
  - 少なくとも「今のファイル」には本物のキーが残っていないか、上記の「何かヒットした場合」と同様に該当ファイルを確認してください。

---

## ステップ5: もう一度スクリプトを実行して確認（任意）

修正したあと、もう一度次を実行します。

```powershell
.\scripts\check-secrets.ps1
```

- 本物のキーをすべてプレースホルダーに変えていれば、該当パターンではヒットしなくなるか、プレースホルダーだけのファイルだけがヒットする状態になります。  
- その状態になれば、公開前の「キー検索」としては完了です。

---

## よくあるエラーと対処

| エラー・表示 | 対処 |
|--------------|------|
| `.\scripts\check-secrets.ps1 が読み込めません。このシステムではスクリプトの実行が無効になっています。` | **ステップ2** の実行ポリシーを設定する（`Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned`） |
| `パス 'scripts\check-secrets.ps1' が存在しません。` | カレントフォルダがプロジェクトのルート（BASHOTORI）か確認する。`cd C:\Users\hoikuen\BASHOTORI` で移動してから再実行 |
| スクリプトは動くが、`.env` の履歴チェックで `git` と出る | Git がインストールされ、`git` コマンドが使えるか確認する。問題なければ「.env は履歴に含まれていないようです」と出ればOK |

---

## まとめ

1. PowerShell を開き、`cd` で **BASHOTORI** のルートに移動  
2. （初回のみ）`Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned` でスクリプト実行を許可  
3. `.\scripts\check-secrets.ps1` を実行  
4. ヒットしたファイルで **本物のキー** があればプレースホルダーに置き換え  
5. 必要ならもう一度スクリプトを実行して確認  

ここまでできれば、方法1の「キー検索」は完了です。
